# Conversation Minute accounting and limit UX

Status: Accepted

A Daily Conversation Minute is charged for the wall-clock interval between `turn_started_at` (Hermes accepts the inbound User Message) and `delivered_at` (Telegram returns `200 OK` on the outbound Anchor Agent reply). The charge is recorded as a single delta against the User's current Daily Minute Usage Row; no separate accounting splits "thinking time" from "delivery time" because that split is invisible to the User and brittle to operate. Voice replies are charged identically, from transcription start to `sendVoice 200 OK`. A Failed Turn — any Conversation Turn that aborts before Turn Delivery — is recorded in the `conversation_turn` table with `status=failed` for 30 days of forensics but is not charged; the User received nothing, so they pay nothing. Repeated retries that drive up underlying LLM cost without driving Conversation Minutes are visible in a separate `agent_cost_log` and addressed operationally if abusive patterns appear, not via the User-facing counter.

The counter lives in Postgres. Hermes calls a Supabase RPC `increment_minutes(anchor_user_id, turn_id, delta_seconds)` that performs an idempotent atomic update keyed on `turn_id`, so a retried call from Hermes after a transient failure cannot double-count. A separate Redis layer is rejected: the increment rate is at most a few per minute per User, Postgres handles this trivially, and a second store would only add a synchronization burden. The User's local midnight is computed as `(now() AT TIME ZONE users.timezone)::date`, where `users.timezone` is an IANA zone string autodetected from the browser at signup (`Intl.DateTimeFormat().resolvedOptions().timeZone`) and editable in account settings. Offsets are rejected as the storage format because they ignore DST transitions. Changing the User Timezone freezes the current local-date row in place — already-consumed seconds remain consumed — and the next local date begins under the new zone. The Daily Minute Usage Row is created lazily on the first delivered turn of each User-local date; no cron job is needed to "reset" counters.

Over-limit behavior reconciles the apparent contradiction between ABR-051 ("agent stops responding") and BPR-009 ("no surprises, no mid-conversation gating"). The Gentle Limit Notice is sent exactly once on the turn that crosses the daily allowance. After that, further inbound User Messages are not handed to the LLM; instead, if the most recent Notice or Refresh was more than two hours ago, Hermes sends a Silence Refresh Notice — a cached static string, no LLM call, no minute charge — so the User does not perceive the bot as broken after a nap or a long pause. If the User explicitly asks why responses stopped (e.g. "warum redest du nicht mehr?"), Hermes runs one Limit Explanation Turn per User per local date: a real LLM call that responds with a scripted-but-natural answer, charged as zero minutes and capped to prevent abuse. The webapp `/account/plan` view shows current Daily Minute Usage and reset time at all times, with a Pro upgrade card; this is the explicit, non-chat surface for "why no responses" so the chat stays free of plan-related friction.

Plan downgrades from Pro to Free apply at the next User-initiated Conversation Turn after the Stripe subscription period ends. If a downgrade timestamp lands while a turn is in flight, Downgrade Grace allows that single in-flight turn to deliver under Pro limits even when delivery happens after the downgrade — the User finishes hearing the reply they started. Subsequent turns are evaluated under Free limits. The maximum bleed-over is a few seconds of one extra reply; the alternative (cutting off mid-turn) breaks BPR-009 trust for no meaningful revenue protection.

Out of scope for MVP: per-message token-based budgeting, anti-abuse beyond the existing cost log, partial-credit refunds for failed turns, push-driven daily summaries of remaining minutes, in-chat minute counters (the explicit Plan view in the webapp is the surface), automatic timezone updates from Telegram or geolocation.
