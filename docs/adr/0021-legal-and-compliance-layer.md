# Legal and compliance layer

Status: Accepted

Anchor is a German-targeted B2C subscription product (EUR pricing, DE default per FR-080). It ships four mandatory legal documents before any non-pilot user is admitted: **Impressum** (§5 DDG — the Digitale-Dienste-Gesetz that replaced the TMG in 2024), **Datenschutzerklärung** (GDPR Art. 13/14), **Allgemeine Geschäftsbedingungen (AGB)**, and **Widerrufsbelehrung** for the subscription. German is the **authoritative legal language**; the English locale (FR-080) carries an informational translation that explicitly defers to the German original in case of conflict. Each document is a localized route under the webapp (`/impressum`, `/datenschutz`, `/agb`, `/widerruf`), is reachable from a persistent footer on every page including the landing page (closing the gap flagged in TODO #15), and is server-rendered static content so crawlers and answer engines can index it (consistent with the AEO posture in the global rules). Company-identifying fields in the Impressum (legal name, address, USt-IdNr., Vertretungsberechtigter, contact) are operator-supplied configuration, not invented here; the template ships with named slots that fail loudly at build if unset.

The **14-day right of withdrawal** (Widerrufsrecht, §355 BGB) is handled explicitly rather than left to Stripe, which does not implement it. Because Pro is a digital service that begins immediately on Checkout, German law (§356 Abs. 5 BGB) lets the withdrawal right lapse only if the User has **expressly consented to immediate performance and acknowledged losing the right**. The Stripe Checkout flow therefore presents a required, unchecked acknowledgment ("Ich verlange ausdrücklich, dass Anchor sofort mit der Leistung beginnt. Mir ist bekannt, dass mein Widerrufsrecht mit Beginn der Ausführung erlischt.") before the subscribe button is enabled, and the acknowledgment event is logged with timestamp against the `anchor_user_id`. Without this consent the subscription does not start. If a User withdraws within 14 days without having consented to immediate performance, the operator refunds via the Stripe dashboard (BPR-010) and downgrades the plan. This is recorded in the AGB and the Widerrufsbelehrung.

Anchor ships **cookie-consent-free by deliberately shipping only strictly necessary cookies**. Under §25 TDDDG (the successor to the TTDSG), strictly necessary storage — here, the single HttpOnly session cookie from magic-link auth (ADR-0006) — requires no consent banner. The MVP carries **no analytics, no third-party trackers, no marketing pixels** on the webapp surface; observability (ADR-0018) is server-side only and stores no client-side identifiers. Consequently Anchor displays a short privacy notice and a link to the Datenschutzerklärung, but **no consent gate**. This is a load-bearing constraint, not an oversight: introducing any non-essential client-side storage (e.g. a product-analytics SDK) later re-triggers the §25 TDDDG consent requirement and must be accompanied by a Consent Management Platform — that decision is explicitly deferred and flagged so it cannot be added silently.

A **sub-processor register** is maintained as a versioned document at `/datenschutz` (subprocessors section) and mirrored in the repo at `docs/compliance/subprocessors.md`. The MVP processor set is: **Stripe** (payments, Stripe Payments Europe DPA), **Google** (Calendar OAuth, Google Cloud DPA), **Telegram** (agent transport), **Anthropic and/or OpenAI** (LLM inference for the agent — data-processing terms, EU/zero-retention options elected where available), **Deepgram with Whisper fallback** (voice transcription per ADR-0014 — the most sensitive flow, EU processing and provider DPA required), **Supabase** (database + Edge Functions, ADR-0010/0013), **Hostinger** (VPS for Hermes, ADR-0003), **Vercel** (webapp hosting, ADR-0011), and **Resend** (transactional email, ADR-0017). Each entry records purpose, data categories, location of processing, and the AVV/DPA reference. Signing the AVV with each processor is an operational prerequisite to launch and is tracked as a checklist in `docs/compliance/`, not as code.

**GDPR data-subject rights are implemented as first-class flows, closing TODO #23.** *Right to erasure (Art. 17):* `/account/delete` lets the User delete the entire Anchor Account self-service. Deletion cascades across the Supabase schema (ADR-0013) — profile, Telegram identity mapping, Google connection (with token revocation against Google), Trusted Person rows, conversation transcripts, diary entries, memory facts and memory-index items, calendar additions, and usage counters — and fires Stripe subscription cancellation. A `deletion_request` row drives a two-step flow: confirmation by magic link, then a 7-day grace window (recoverable, since this audience makes accidental requests) after which a scheduled job (ADR-0019) performs the irreversible purge and writes a tombstone audit event. *Right to export (Art. 20):* `/account/export` generates a machine-readable JSON bundle (profile, diary, memory facts, transcripts, calendar additions) plus a human-readable summary, delivered as a download link emailed via Resend; generation is asynchronous to avoid request-timeout on large memory sets. Both flows have an operator fallback (ADR-0018) for users who have lost webapp access, and both are reachable from the Datenschutzerklärung as required by Art. 13(2)(b).

These decisions update the requirement web: NFR-008's "documented before production launch" is discharged for erasure/export by this ADR and the new `compliance-requirements.md` leaf; the Open Questions doc gains an explicit consent/legal resolution; and the landing-page footer requirement (WUX) gains the legal links.

Draft content for two of the four surfaces now exists in-repo as the source-of-truth German copy: `docs/compliance/impressum.md` and `docs/compliance/datenschutz.md`, adapted from the sibling Empatify product's live pages. They fix the concrete operator identity — **Rudolpho AI – Marco Rudolph, No de Halloh 8a, 25591 Ottenbüttel, USt-IdNr. DE455180377** (§27a UStG) — and the concrete sub-processor disclosures (Telegram, Google Calendar, Anthropic/OpenAI, Deepgram+Whisper, Stripe, Resend, Supabase, Vercel, Hostinger), and they encode the cookie-consent-free posture verbatim. Section structure mirrors the Empatify `page.tsx` so each converts 1:1 to `src/app/[locale]/{impressum,datenschutz}/page.tsx` once the Next.js app (ADR-0011) is scaffolded. What this ADR still does **not** do is finalize the AGB and Widerrufsbelehrung prose, nor obtain qualified legal review — the draft Impressum/Datenschutz copy and the not-yet-written AGB/Widerruf are reviewed with a qualified party against the operator's real company data before launch (CR-011); this record fixes the architecture, the required surfaces, the consent mechanics, and the data-subject-rights implementation so engineering can build against a stable contract.

Out of scope for MVP: multi-jurisdiction legal variants beyond DE/EU, a Consent Management Platform (deferred with the no-non-essential-cookies constraint above), automated DSAR (data-subject access request) ticketing beyond the self-service export, age-verification gating, a separate B2B/AVV-as-controller offering, accessibility-conformance legal certification (BFSG) — tracked separately, not blocked here.
