# Anchor uses a domain-specific memory index instead of OB1 thoughts

Anchor will use OpenBrain's Supabase, pgvector, and Edge Function patterns, but not OB1's generic `thoughts` table as the product memory index. Anchor needs a domain-specific memory index linked to canonical Conversation Transcripts, Diary Entries, Memory Facts, Calendar Events, and Recall Exclusion state, with explicit `anchor_user_id`, `source_kind`, nullable foreign keys such as `conversation_transcript_id`, `diary_entry_id`, `memory_fact_id`, or `calendar_event_id`, `occurred_at`, `recall_excluded_at`, and domain metadata. This avoids terminology and migration conflicts with OB1 while preserving the useful OpenBrain architecture: canonical records in Postgres, full-text/vector indexes for retrieval, and Edge Functions that enforce permissions and recall rules. Anchor deliberately avoids free-form `source_table`/`source_id` strings for canonical source links because real foreign keys provide better integrity, cascades, and query planning.

The memory index schema must enforce source integrity with a CHECK constraint: exactly one canonical source foreign key is populated, and `source_kind` must match that populated foreign key. This prevents ungrounded index rows and ambiguous recall citations.
