# Voice transcription: modular Deepgram with Whisper fallback

Status: Accepted

Anchor transcribes Telegram voice messages before passing them to the LLM. The primary provider is **Deepgram Nova-3** (German `de` language code, batch mode) and the secondary provider is **OpenAI Whisper** (`whisper-1`). Both providers sit behind a single `Transcriber` interface in the Hermes codebase exposing one method `transcribe(audio: Buffer, lang: 'de' | 'en'): Promise<{ text: string; durationMs: number; provider: string; cost_cents: number }>`. The active provider is selected per call by a small `pickTranscriber()` factory that reads an env-configured primary plus health state (consecutive failures, circuit-breaker open/closed) and routes the next call accordingly. Swapping in a future provider — Gemini Audio Native, a self-hosted Whisper on a GPU VPS, anything else — is one new implementation of `Transcriber` plus one entry in the factory.

Deepgram was chosen as the primary because Nova-3's measured German accuracy on Telegram-quality microphones (~16 kHz Opus) is at the top of the current field, latency is one-to-two seconds for typical voice-note durations, and the per-minute price (~$0.0043/min as of 2026) keeps the per-Pro-User monthly voice cost in single-digit cents at expected usage. Whisper was chosen as the fallback because it is stable, well-documented, and the per-minute price (~$0.006/min) is acceptable for sporadic failover traffic. Gemini Audio Native was considered and deferred: it removes the separate transcript step entirely (audio goes directly into the LLM call), which is architecturally interesting, but it ties the transcription quality to the chosen LLM and is harder to A/B-evaluate; revisit post-MVP if voice volume justifies. AssemblyAI was considered and rejected because its German quality lags the alternatives. Local Whisper on the Hostinger VPS is rejected for MVP because the VPS is CPU-only and CPU Whisper latency violates NFR-005; revisit when a GPU host becomes economical (~300+ active voice-using Users).

Audio data lifecycle: a Telegram voice file is downloaded into Hermes memory, sent to the active Transcriber over HTTPS with the provider's training-opt-out flag enabled, and the raw audio is discarded as soon as the transcript returns. No audio is persisted in Anchor or in Hermes. Provider-side retention is configured to the minimum each provider supports (Deepgram: "do not store" header; OpenAI: API calls are not used to train models, retained 30 days for abuse monitoring then deleted, which is the platform default and we accept it). The resulting transcript is written into `conversation_turn.transcript` in the Anchor Postgres on the Hostinger VPS, which is the only persistent home for voice-derived content. Standard SCCs cover the EU→US transfer; this is documented in the public sub-processor list (separate legal-layer work).

Voice handling is **opt-in but enabled by default at signup**. The onboarding wizard includes an explicit voice-consent checkbox with German copy approximately: _"Ich möchte Sprachnachrichten an Anchor senden können. Diese werden zur Texterkennung an einen externen Dienstleister übertragen (Deepgram, USA, Standardvertragsklauseln) und nach der Verarbeitung gelöscht."_ A User who unchecks the box gets a text-only experience; the Anchor Agent declines voice messages with a friendly prompt to type. Users may toggle the consent later in `/account/privacy`. The consent state is stored in `anchor_user.voice_consent_at TIMESTAMPTZ` so that revoking the consent has a clear effective timestamp for audit purposes. Voice is treated conservatively as potentially Article-9 GDPR data even where the technical biometric threshold may not be crossed.

Cost is tracked per transcription call in `agent_cost_log` with columns `provider`, `duration_ms`, `cost_cents`, `anchor_user_id`, `turn_id`. A monthly per-User cost cap (default: €5 of voice-provider cost) raises an operational alert and, if exceeded, throttles further voice calls for that User for the remainder of the month with a Gentle Limit-style notice. The latency surface stays well inside NFR-005's 20-second budget: Deepgram ~1.5s + LLM ~5s + Telegram send ~0.5s leaves headroom.

Out of scope for MVP: streaming transcription, real-time voice replies (text-to-speech of Anchor's answer is a separate decision), multi-language auto-detect (the User's locale fixes the transcription language), on-device transcription, audio caching for diary replay.
