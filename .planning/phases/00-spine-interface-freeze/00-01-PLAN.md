<objective>
Prove the self-hosted Supabase environment on the real Hostinger VPS can support the locked architecture BEFORE any schema is frozen. This is the fail-fast gate for the dominant Phase-0 risk (Pitfall 1: self-hosted Supabase ≠ Supabase Cloud) and the beta self-hosted Edge Functions / extension risk.

Per the locked decision, this is the FIRST task of the phase. It runs a `CREATE EXTENSION` smoke migration against the live VPS Postgres to prove `pg_uuidv7`, `vector`, `pg_cron`, `pg_net`, and `pgcrypto` install, and verifies the `supabase_realtime` publication + `REPLICA IDENTITY` mechanism works. If `pg_uuidv7` is unavailable, the fallback (JS `uuidv7` 1.2.1 npm, default applied app-side) is selected and recorded so the schema-freeze plan (00-03) knows which UUIDv7 strategy to use.

Purpose: De-risk the freeze. The frozen DDL in 00-03 hard-depends on knowing which extensions are real on this VPS.
Output: A smoke migration, a psql migration runner, and a handoff record of the resolved extension/uuidv7/realtime strategy.
</objective>

After the extension block, add a probe that FAILS LOUDLY if uuidv7 is missing — `do $$ begin perform uuid_generate_v7(); exception when undefined_function then raise exception 'pg_uuidv7 unavailable — use uuidv7 npm fallback (see docs/handoff.md)'; end $$;`. Then a realtime probe: create a throwaway table `_rt_smoke(id uuid primary key default gen_random_uuid())`, `alter table _rt_smoke replica identity full;`, `alter publication supabase_realtime add table _rt_smoke;`, then `alter publication supabase_realtime drop table _rt_smoke;` and `drop table _rt_smoke;` so the probe leaves no residue. The migration must be idempotent/re-runnable.

    Do NOT freeze any real schema here — that is 00-03. This file is a disposable probe.
