- Hermes (DEC-0002/0003): separate Docker container on the VPS, calls edge fns over loopback (warm, sub-ms). ADRs do NOT pin Hermes' internal framework — discretion: keep it thin. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram servers → Hermes webhook | Untrusted update payloads enter Hermes |
| Hermes (service-role + x-hermes-secret) → Edge Function | RLS bypassed; explicit handler authz |
| Browser Realtime channel → anchor_user row | Per-user scoped subscription |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-21 | Spoofing | Wrong Telegram identity bound to an account | mitigate | Code is bound to the issuing account at mint; pairing-redeem writes telegram_user_id only for that account; `telegram_user_id` UNIQUE constraint (00-03); the E2E asserts no cross-account binding. |
| T-00-22 | Spoofing / EoP | Pairing-code brute force / stolen-session takeover | mitigate | ~40-bit Crockford code + 15-min TTL + single-use atomic burn + 5/hr issuance rate-limit (DEC-0007); Hermes validates code format before redeem. |
| T-00-23 | Information Disclosure | Cross-account leak via Realtime channel | mitigate | `useConnectionStatus` subscribes only to `anchor_user WHERE id = auth.uid()` (RLS-scoped); no cross-user fan-out (DEC-0015). |
| T-00-24 | Tampering | Forged Telegram webhook payload | mitigate | Telegram webhook secret-token / bot-token validation in `webhook.ts`; only well-formed `/start` updates reach redeem; redeem itself is idempotent + TTL/single-use bounded. |
| T-00-25 | Information Disclosure | Optimistic "connected" before the server write | mitigate | The badge binds strictly to the `anchor_user` DB row via Realtime (no optimistic render); BPR-009 "no surprises" upheld. |
</threat_model>
